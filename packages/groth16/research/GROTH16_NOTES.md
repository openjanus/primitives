# Groth16 — Research Notes

## What is Groth16?

Groth16 is a zero-knowledge succinct non-interactive argument of knowledge (zk-SNARK)
protocol introduced by Jens Groth in 2016. It is the most widely deployed ZK proof
system in production applications (Tornado Cash, Zcash Sapling, etc.).

Key properties:
- **Succinct**: Proof size is constant (~200 bytes) regardless of statement complexity.
- **Non-interactive**: No back-and-forth between prover and verifier.
- **Zero-knowledge**: The proof reveals nothing beyond the fact it is valid.
- **SNARK**: Proof can be verified much faster than computing the witness directly.

## How it works (high level)

1. **Circuit**: Express the computation as arithmetic constraints (R1CS: rank-1 constraint system).
2. **Trusted setup**: Generate a structured reference string (SRS) for the specific circuit.
   This is the "toxic waste" ceremony — if compromised, fake proofs become possible.
3. **Prove**: Given private inputs (witness), compute a proof `(π_A, π_B, π_C)` in G1/G2.
4. **Verify**: Check the bilinear pairing equation:
   `e(π_A, π_B) = e(α, β) * e(vk_x, γ) * e(π_C, δ)`
   where `vk_x` is a linear combination of the public inputs.

## BN254 and precompiles on Flow EVM

The underlying group is BN254 (also called alt_bn128 or bn128), which supports
efficient pairing operations. Flow EVM implements the EIP-197 precompiles:

| Precompile | Address | Operation | Gas |
|------------|---------|-----------|-----|
| `ecAdd` | `0x06` | G1 point addition | 150 gas |
| `ecMul` | `0x07` | G1 scalar multiplication | 6,000 gas |
| `bn128pairing` | `0x08` | BN254 pairing check | 45,000 + 34,000*n |

The Groth16 verifier uses all three:
- `ecMul` for computing `IC_i * pubSignals[i]` (linear combination of public inputs)
- `ecAdd` for accumulating the linear combination
- `bn128pairing` for the final pairing check

For the ConfidentialTransfer circuit with 6 public signals, the gas cost is:
- 6 × ecMul: 36,000 gas
- 6 × ecAdd: 900 gas
- bn128pairing (4 pairs): 45,000 + 4 × 34,000 = 181,000 gas
- **Total: ~253,531 gas** (measured on Flow EVM testnet)

## The pi_b Fp2 swap

This is the most common source of Groth16 verification failures on EVM.

snarkJS emits `pi_b` in `(real, imaginary)` order for each Fp2 element:
```javascript
proof.pi_b = [
  [re_0, im_0],  // snarkjs order
  [re_1, im_1],
]
```

But EIP-197 (the `bn128pairing` precompile) expects `(imaginary, real)`:
```
EVM expects: [[im_0, re_0], [im_1, re_1]]
```

This swap MUST be applied before calling the on-chain verifier:
```javascript
const pB = [
  [proof.pi_b[0][1], proof.pi_b[0][0]],  // swap within each Fp2 element
  [proof.pi_b[1][1], proof.pi_b[1][0]],
];
```

This requirement comes from EIP-197's encoding of Fp2 elements:
> "Fp2 elements are encoded by their two components in big endian form:
> `a * i + b` is encoded as `a, b` (big endian)"

snarkJS uses the opposite convention internally. The Solidity verifier contract
(generated by snarkJS) expects inputs with the swap already applied.

## Trusted setup

Groth16 requires a circuit-specific trusted setup (phase 2). The universal
structured reference string (e.g., Hermez Powers of Tau) provides the
general setup (phase 1), and then a circuit-specific ceremony generates
the proving key and verification key.

For the ConfidentialTransfer circuit in this repo:
- **Phase 1**: Hermez pot28 (multi-party, large community ceremony)
- **Phase 2**: Single-contributor contribution (experimental/lab use only)
- **Risk**: If the phase 2 contributor saved their "toxic waste", they could
  generate fake proofs for any witness. DO NOT use in production.

## Verification key structure

The verification key (vk) contains:
- `alpha` (G1): From the trusted setup
- `beta` (G2): From the trusted setup
- `gamma` (G2): Generator-related constant
- `delta` (G2): From the circuit-specific setup
- `IC` (G1 array): One per public input + 1 for the constant term

The deployed ConfidentialTransferVerifier has these hardcoded as Solidity constants,
generated by `snarkjs generateverifier`. Redeploying with different constants
would create a different verifier for different circuits or setups.
